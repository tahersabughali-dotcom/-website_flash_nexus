# Apply Migrations to Development Supabase

**Development only. Never production.**

---

## Source of Truth

| Path | Role |
|------|------|
| `database/migrations/` | **Canonical** SQL — edit here |
| `supabase/migrations/` | CLI copy — generated by sync script |
| `database/seeds/` | Seed data (run after migrations) |

---

## Step 0: Environment (Required)

Create **`flash-nexus/.env.local`** (not the workspace root):

```bash
cd flash-nexus
cp .env.example .env.local
```

Fill in from **Supabase Dashboard → Project Settings → API** (development project only):

- `NEXT_PUBLIC_SUPABASE_URL`
- `NEXT_PUBLIC_SUPABASE_ANON_KEY`
- `SUPABASE_SERVICE_ROLE_KEY`

Verify (does not print values):

```powershell
# Keys must show SET with length > 0
```

`.env.local` is gitignored — never commit it.

---

## Step 1: Link Development Project

One-time setup (interactive):

```bash
cd flash-nexus
npx supabase login
npx supabase link --project-ref YOUR_PROJECT_REF
```

Find `YOUR_PROJECT_REF` in Supabase Dashboard → Project Settings → General → Reference ID.

---

## Step 2: Sync Migrations

```bash
npm run db:sync-migrations
```

Copies `database/migrations/*.sql` → `supabase/migrations/` with CLI timestamps.

---

## Step 3: Apply Migrations (CLI — Recommended)

```bash
npm run db:push
```

Or separately:

```bash
npm run db:sync-migrations
npx supabase db push
```

Applies only to the **linked** development project.

---

## Alternative: SQL Editor (Manual)

If CLI is unavailable:

1. Open Supabase Dashboard → SQL Editor (dev project).
2. Run each file from `database/migrations/` in order **001 → 010**.
3. Wait for success before the next file.
4. Run `database/seeds/001_seed_roles_permissions.sql`.

## Step 6: Generate TypeScript Types

After schema is applied:

```bash
npm run db:types
```

Writes generated types to `lib/supabase/types.ts`.

Requires linked project (`supabase link`).

---

## Future Migrations

1. Add new file to `database/migrations/` (next number).
2. Run `npm run db:sync-migrations`.
3. Test on **development** with `npm run db:push`.
4. Run smoke tests.
5. Run `npm run db:types`.
6. Never apply untested migrations to production.

---

## Production Warning

Do **not** run `db push` against production until:

- Staging validation complete
- Backup verified
- Documented in `docs/DEPLOYMENT_AND_VERSIONING.md`
